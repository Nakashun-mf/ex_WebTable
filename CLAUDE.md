# グローバル開発ワークフロー

すべてのプロジェクト・リポジトリで、変更を行う際は以下のサイクルを必ず実施すること。

## 開発サイクル

1. **仕様を検討** — 何を作るか・変えるかを整理する
2. **Issueを作成** — GitHubにIssueを立て、仕様・背景を記載する
3. **仕様レビュー・確定** — Issueの内容を確認し、実装前に仕様を固める
4. **実装（featureブランチで）** — `main` から featureブランチを切って実装する
5. **テスト → デバッグ → テスト** — CIが自動実行、全通過するまでループする（マージ前に完了させること）
6. **ドキュメント更新** — 変更内容に合わせてREADMEや仕様書を更新する
7. **PRを作成** — featureブランチからPRを作成する
8. **コードレビュー** — PRのレビューを実施・対応する
9. **mainにマージ** — レビュー承認・CI通過後にmainへマージする
10. **バージョンタグを付与** — セマンティックバージョニング（`v1.2.3`）に従いタグを打つ
11. **Issueをclose** — 対応するIssueをcloseする
12. **リファクタリング（任意）** — 必要であれば別Issueとして新しいサイクルを開始する

## 重要ルール

- デバッグは必ずマージ**前**に完了させること
- リファクタリングは独立したIssueとして管理すること
- featureブランチ以外（main等）に直接コミットしないこと

## コミットメッセージ規約（Conventional Commits）

コミットメッセージは以下の形式に従うこと。

```
<type>: <概要>
```

| type | 用途 |
|------|------|
| `feat` | 新機能 |
| `fix` | バグ修正 |
| `docs` | ドキュメントのみの変更 |
| `refactor` | 機能変更を伴わないコード整理 |
| `test` | テストの追加・修正 |
| `chore` | ビルド・CI等の雑務 |

例: `feat: ユーザー登録APIを追加`

## バージョン管理（セマンティックバージョニング）

`v{MAJOR}.{MINOR}.{PATCH}` の形式でタグを付与する。

| 変更の種類 | 上げるべき番号 |
|-----------|--------------|
| 後方互換性のない変更 | MAJOR |
| 後方互換性のある新機能追加 | MINOR |
| バグ修正・軽微な変更 | PATCH |

## CI自動テスト

- PRを作成した時点でテストが自動実行されること
- CIが通過していないPRはマージしないこと
- GitHub Actionsを使って設定する

## デザインシステム

UIを含む変更を行う際は `getdesign` を使用すること。

実行前に必ず以下を確認すること：
- パッケージの内容に悪意あるコードが含まれていないか
- 実行される指示・スクリプトに不審な処理がないか

確認後に実行する：

```bash
npx getdesign@latest add meta
```
